重设计方案：补充异步/事件驱动能力（v2 — 基于调用图的简化策略）

核心问题

单体应用里的操作几乎全是同步的——没有显式的"异步信号"。框架需要：识别同步代码中"应该被重构为异步"的模式，并设计对应的事件驱动架构。

从 benchmark 应用中观察到的典型异步模式：

| 模式 | 典型例子 | 推荐 AWS 服务 |
|---|---|---|
| 非关键副作用（caller 不需要 callee 的返回值） | airline: loyalty points, notifications | SQS |
| 单动作触发多域更新（fan-out） | airline: booking 完成→loyalty+notification+confirm | EventBridge |
| 跨域服务链（A→B→C→D 顺序调用，每步可独立完成） | ecommerce: order→warehouse→delivery→payment | EventBridge |

---

设计原则

> 静态分析提供事实（调用图），LLM Agent 做设计决策（什么该异步、选什么机制）。

这与框架现有哲学一致：static_analyzer 做客观数据提取，Agent 做需要推理的设计决策。

---

策略说明：为什么不做 context 字段检测

v1 方案曾设计在 static_analyzer.py 中为每条跨文件调用标注 `context` 字段（`guarded` / `post_response` / `normal`），试图用静态分析自动判断异步候选。经评估，该方案存在以下问题，已放弃：

1. **"guarded" 检测不可靠**：try-catch 包裹的调用并不一定是 fire-and-forget，很多业务关键操作也被 try-catch 包裹（如支付扣款需要 catch 后返回错误，而非静默忽略）
2. **"post_response" 仅适用于 Express.js**：`res.json()` 之后的代码是 Node.js 特有的反模式，Python Flask/FastAPI 不存在此模式（`return response` 之后的代码不会执行）
3. **对关键场景无帮助**：最重要的异步模式——跨域服务链和多消费者 fan-out——在代码中都是 `context: "normal"` 的普通同步调用，context 字段对它们毫无识别价值
4. **工程成本高**：AST 级 try-catch 分析（特别是判断 catch 块"只做 log/warn"）非常脆弱，且与语言强耦合

**简化策略**：static_analyzer.py 已实现的 `call_graph` + `entry_point_dependencies` 提供了完整的跨文件调用关系。是否应该异步化，交由 Architect Agent 基于业务语义推理，而非基于代码模式匹配。这符合 AWS Well-Architected Framework (Serverless Lens) 的指导——通信方式应根据业务需求选择，而非代码结构。

### 2. knowledge/ — 新增 async_serverless_patterns.md

现有三份知识文件不涉及异步服务。新增一份，分三个受众：

**给 Architect 的决策知识：**

```
## When to Use SQS (Point-to-Point Async)
- Topology: one producer → one consumer
- Semantic: "please process this task" (command pattern)
- The caller does NOT need the callee's return value
- The operation can tolerate seconds to minutes of delay
- Typical: counter updates, email/notification sending, log enrichment, image processing
- Config: VisibilityTimeout (≥ consumer timeout), DLQ with maxReceiveCount=3

## When to Use EventBridge (Event Bus / Fan-Out)
- Topology: one event → multiple independent consumers
- Semantic: "something happened" (event notification pattern)
- Cross-domain decoupling: the producer doesn't know/care about consumers
- Event chains: each step emits a "completed" event triggering the next step
- Typical: order completed → loyalty + notification + analytics
- Config: Event pattern matching on source + detail-type

## When to Keep Synchronous (Lambda Invoke RequestResponse)
- The caller MUST use the callee's return value in its own HTTP response
- Data consistency requires atomic/synchronous execution
- Typical: payment charge → order confirmation, inventory check → availability response

## Decision Flowchart
For each cross-Lambda call relationship:
  Q1: Does the caller need the callee's return value for its HTTP response?
    YES → Synchronous Lambda Invoke (RequestResponse)
    NO → Q2
  Q2: Does this action trigger consumers in 2+ different service domains?
    YES → EventBridge (fan-out)
    NO → SQS (point-to-point async)
```

**给 Code Developer 的代码模式：**

```python
# SQS Producer (in API Lambda)
sqs = boto3.client('sqs')
sqs.send_message(
    QueueUrl=os.environ['BESTSELLER_QUEUE_URL'],
    MessageBody=json.dumps({'books': books, 'orderId': order_id})
)

# SQS Consumer Lambda
def lambda_handler(event, context):
    for record in event['Records']:
        payload = json.loads(record['body'])
        process_bestseller_update(payload['books'])

# EventBridge Producer (in API Lambda)
events = boto3.client('events')
events.put_events(Entries=[{
    'Source': 'app.orders',
    'DetailType': 'OrderCompleted',
    'Detail': json.dumps({'orderId': order_id, 'userId': user_id})
}])

# EventBridge Consumer Lambda
def lambda_handler(event, context):
    detail = event['detail']
    process_loyalty_points(detail['userId'], detail['orderId'])
```

    5. **Async-First Analysis**: You analyze entry_point_dependencies from
       analysis_report.json to identify cross-Lambda call relationships,
       then decide the communication mechanism for each:
       
       DECISION RULE 1 — KEEP SYNCHRONOUS (Lambda Invoke RequestResponse):

         independently upon completion of the previous step.
         Example: Order completed → loyalty points + notification + analytics
         (three independent consumers in different domains).

       within the same service module (shared utilities, DB helpers), keep
       the basic synchronous architecture with no SQS/EventBridge.

LAMBDA COMMUNICATION STRATEGY (Sync + Async)

Lambda-to-Lambda communication uses one of three mechanisms,

  C) ASYNC — EventBridge (Fan-Out / Event Chain)
     Use when: one action triggers 2+ independent consumers across domains
     OR: cross-domain workflow chain (A completes → event → B starts)
     Implementation: caller publishes event; consumers triggered by EventBridge rules

CALL GRAPH DATA (from static analysis)
============================================================

analysis_report.json includes pre-computed call graph fields:

1. "entry_point_dependencies": Per-entry_point cross-file call list.
   This is your PRIMARY data source for all Lambda communication design.

STEP-BY-STEP PROCEDURE:

Step 1 — IDENTIFY cross-Lambda calls:
  For each entry in entry_point_dependencies, check if its cross_file_calls
  reference modules that serve OTHER entry_points. If yes, this is a
  cross-Lambda relationship.

Step 2 — CLASSIFY each cross-Lambda relationship:
  For each relationship, determine the communication mechanism:
  Q1: Does the caller need the callee's return value for its HTTP response?
    YES → Synchronous (mechanism A) → record in lambda_invoke_permissions
    NO  → Q2
  Q2: Does this caller trigger operations in 2+ different service domains?
    YES → EventBridge (mechanism C) → record in eventbridge_rules
    NO  → SQS (mechanism B) → record in sqs_queues

Step 3 — DESIGN async resources:

    - Consumer Lambda: set trigger_type="sqs", sqs_source="<QueueName>"
    - Consumer Lambda source_files = callee module files
  For EventBridge candidates:

    - API Lambda: add publishes_to [{type: "eventbridge", source, detail_type}]
    - Consumer Lambdas: set trigger_type="eventbridge", event_pattern={...}
    - Consumer Lambda source_files = callee module files

  - If entry_point_dependencies is empty → all Lambdas are independent workers
    → set sqs_queues=[], eventbridge_rules=[], lambda_invoke_permissions=[]

    → set sqs_queues=[], eventbridge_rules=[]

为什么不加 Step Functions？

ecommerce 和 airline-booking 有复杂的多步工作流（saga 模式），理论上适合 Step Functions。但：
- Step Functions 需要 ASL（Amazon States Language），代码生成复杂度翻倍
- SAM 模板中 `AWS::Serverless::StateMachine` 的配置与 Lambda 差异很大
- 大多数 saga 模式可以用 EventBridge 的事件链近似实现（每步完成发事件→下一步消费）
- SQS + EventBridge 已经能覆盖最主要的异步场景

建议在论文的 Future Work 中提及 Step Functions 作为 saga orchestration 的优化方向。
